# 分层总图说明

这一版只回答三个问题：

1. 谁是入口？
2. 谁是真正的大脑？
3. 谁保存真实状态？

## 结论

- **用户命令层**：只发动作
- **workflow-engine.ts**：唯一编排核心
- **workflow/ 文件层**：唯一事实来源
- **isolated pi worker**：只执行单节点
- **UI / board**：只展示与人工干预

## 为什么上一版会乱

因为把三件事混在一起画了：
- 系统分层
- 单节点执行流程
- 状态模型

这三种视角都对，但不该塞进一张主图。

## 正确阅读顺序

先看这张分层总图，明确边界。
之后如果还要继续，再分别看：
- 单节点执行时序图
- 文件状态模型图
- 默认 pipeline 图

## 核心定位

这个插件当前更像：
- 有状态的串行 phase runner
- 文件驱动 workflow runtime
- 带审批门和重试机制的单节点 worker 编排器

它当前还不像：
- 通用 DAG 调度器
- 并行拓扑执行引擎
- 大规模任务队列系统
